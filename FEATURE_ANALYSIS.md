## 1. TL;DR

- **Ядро работает**: создание турнира, публичная регистрация по slug, single-elimination сетка, ручная drag-drop расстановка, ввод счёта по сетам, роли owner/editor, 3 языка, dark theme, realtime.
- **Заявлено, но НЕ реализовано** (расхождения docs ↔ код): live-scoring по очкам (нет `useTennisScoring.js` и таблицы `live_scores`), роль `counter`, профили игроков с историей, связка клуб↔турнир (`tournaments.club_id` не используется в `create_tournament`).
- **Главная архитектурная дыра**: нет сущности `players` как first-class — всё хранится в `entry_members` как строки. Без этого невозможны профили, ELO, история матчей.
- **Клубный пивот начат, но не завершён**: таблицы `clubs` и `platform_admins` есть, super-admin одобряет клубы — но клубы не владеют турнирами и не имеют публичной витрины.
- **Топ-5 рекомендаций на 3–6 месяцев**:
  1. Закрыть P0-долг (секция 3) — иначе дальнейшие фичи строятся на песке.
  2. Публичная страница клуба `/clubs/:slug` + связать турниры с клубами.
  3. Round-robin и **group stage + playoff** (второй — в FEATURE_RESEARCH отсутствует).
  4. Категории участников (возраст/пол/уровень) и waiting list — ожидание любого серьёзного клуба.
  5. Telegram-бот — самая дешёвая killer-фича для RU/LT/EE рынка.

## 3. P0 — технический долг и блокеры роста

Эти пункты нужно закрыть **до** любых новых фич, иначе последующие фичи будут либо дублировать фундамент, либо разваливать старый.

### 3.1. Live-scoring: либо делаем, либо убираем из docs
Расхождение между декларацией и реальностью. Три варианта:
1. **Реализовать**: создать `src/lib/useTennisScoring.js` (композабл с состоянием points/games/sets/tiebreak/deuce/undo-stack), таблицу `live_scores`, RPC `start_live_match()`/`stop_live_match()`/`record_point()`. Подключить к существующим `LiveScoringModal.vue` / `LiveScoreViewerModal.vue`. **≈ 3–5 дней**.
2. **Убрать из docs**: честно написать в PRODUCT.md, что live-scoring — в roadmap, а не в MVP.
3. **Временный компромисс**: использовать live-scoring как «ввод геймов по очереди» без логики точек. Дёшево, не вводит в заблуждение.

**Рекомендую вариант 1** — компоненты уже написаны, это killer-фича по маркетингу («realtime по очкам»).

### 3.2. Связать клубы с турнирами
В `create_tournament()` добавить параметр `p_club_id` и заполнять `tournaments.club_id`. Если пользователь — владелец клуба, турнир по умолчанию создаётся под этим клубом. Без этого `/clubs/:slug` витрины не будет.

### 3.3. Ввести `players` как first-class сущность
Сейчас имена участников — строки в `entry_members.full_name`. Для истории/рейтинга/матчмейкинга нужна отдельная таблица:
```
players (id, contact_hash, display_name, created_at, merged_into uuid null)
entry_members.player_id → players.id
```
Ключ склейки — нормализованный телефон/email (`contact_hash`). При регистрации через `register_entry()` ищется существующий player, иначе создаётся. Добавить админский инструмент «объединить дубликаты» (поле `merged_into`).

### 3.4. Синхронизировать CLAUDE.md со схемой
- Убрать упоминание `counter` роли или добавить её в БД.
- Убрать упоминание `useTennisScoring.js`, пока не реализован.
- Добавить упоминание `clubs` и `platform_admins` (их в CLAUDE.md нет).

### 3.5. Защита от спама публичной регистрации
`register_entry()` вызывается без авторизации. Сейчас rate limiting нет — один бот может создать 10 000 заявок. Варианты: captcha на форме, rate limit по IP через Supabase Edge Function, обязательная валидация телефона через SMS-код (позже).

## 4. Альтернативные форматы турниров (главный блок новизны)

Сейчас в `bracket_format` только `single_elimination`. Это критическое ограничение: клубы регулярно запрашивают другие форматы. Ниже — полный список с оценкой сложности и B2B-ценности.

### 4.1. Сводная таблица

| Формат | Аудитория | Сложность | Уже в FEATURE_RESEARCH? |
|---|---|---|---|
| **Round-robin (группы)** | 4–8 игроков, клубные вечера | M | Да (Фаза 2, #8) |
| **Group stage + playoff** | 16–32, формат ATP 250 | L | **Нет** |
| **Double elimination** | Серьёзные турниры с losers bracket | L | Да (Фаза 4, #13) |
| **Swiss system** | 6–9 туров без вылета | M | **Нет** |
| **Consolation bracket** | Проигравший 1-й круг играет утешительный | M | **Нет** |
| **Ladder / Box league** | Постоянная клубная лига с challenge-матчами | L | **Нет (сильная B2B-фича)** |
| **Compass draw** | Юниоры, все играют ≥3 матча | M | **Нет** |
| **Team format** | Клуб vs клуб, сумма очков пар/одиночек | L | **Нет (ключевая B2B)** |
| **Multi-stage template** | Этап 1 RR → этап 2 SE (комбинация) | L | **Нет** |

### 4.2. Детали по каждому формату

**Round-robin (группы)**
Каждый играет с каждым. Победитель — по W/L, потом по разнице сетов/геймов. В БД: `bracket_format = 'round_robin'`, `groups` таблица, все пары генерируются разом. UI — standings-таблица с live-обновлением.

**Group stage + playoff (самый востребованный в клубах)**
Этап 1: 4 группы по 4 человека, round-robin. Этап 2: top-2 из каждой группы → single-elim на 8 (четвертьфинал). Нужна схема «этапов» в турнире: `tournament_stages (id, tournament_id, order, format)`. Переход между этапами — кнопка «Завершить групповой этап → сгенерировать плей-офф».

**Swiss system**
N туров, пары формируются по близости очков после каждого тура, повторные пары запрещены. Нужна функция `generate_swiss_round(tournament_id, round_number)` — учитывать матрицу «кто с кем играл». Актуально для падел-клубов и мастерсов без вылетов.

**Consolation bracket**
Параллельная сетка для проигравших 1-го круга. Тривиально на уровне схемы (второй корень `matches`), но нужна UI-видимость («основная сетка / утешительная»).

**Ladder / Box league** ⭐ клубная фича
Постоянная лига внутри клуба без жёстких дат. Игроки делятся на «коробки» по 4–6 человек, каждую неделю играют 1–2 матча с другими в своей коробке. В конце месяца top-2 поднимаются в более высокую коробку, bottom-2 опускаются. **Это то, ради чего клубы платят за год подписки.** БД: `leagues`, `league_boxes`, `league_challenges` (вызов на матч), периоды (сезоны).

**Compass draw**
Специфичный юниорский формат. Первый круг — single-elim. Проигравшие распределяются в «север/юг/запад/восток» — мини-сетки. Каждый играет минимум 3 матча. Важно для детских турниров.

**Team format (клуб vs клуб)** ⭐ B2B-ценность
Матч между клубами — это N одиночных + M парных. Результат команды = сумма победителей. Нужны сущности `teams`, `team_members`, `team_matches` с ссылками на обычные `matches`. Открывает межклубные турниры и лиги — сильный драйвер удержания клубов на платформе.

**Multi-stage template**
Турнир как последовательность этапов разного формата. Пример: «Летний кубок 2026» = этап-1 (round-robin по группам) → этап-2 (single-elim из top-2 групп) → финал (best-of-5). Переиспользует сущности из других форматов, но нужна UI-оркестровка.

### 4.3. Приоритизация форматов на 3–6 месяцев
Не делать всё сразу. Порядок:
1. **Round-robin** (нужен всем, минимум кода).
2. **Group stage + playoff** (самый востребованный у клубов, переиспользует RR).
3. **Ladder / Box league** (клубная удерживающая фича, отдельный эпик).
4. **Team format** (когда есть хотя бы 5+ активных клубов).

Double elimination, Swiss, consolation, compass — отложить за горизонт 6+ месяцев.

## 5. Фичи, которых нет в FEATURE_RESEARCH.md

Сгруппированы по теме. В каждой подсекции — от 3 до 6 пунктов с указанием, для кого ценно.

### 5.1. Клубная витрина (новое, под гибридную модель)

- **Публичная страница клуба** `/clubs/:slug`: описание, контакты, карта, календарь турниров, история результатов. Главная монетизационная точка клуба.
- **Членство в клубе** (подписка/абонемент): приватные турниры только для членов, скидки на entry fee. Таблица `club_members (club_id, user_id, tier, expires_at)`.
- **Embed-виджет сетки для сайта клуба**: `<iframe src="…/embed/:slug">` одной строкой. Клубам важно показывать турнир на своём сайте, а не уводить трафик.
- **Клубный рейтинг** (ELO внутри клуба, независимо от глобального): клубы гордятся внутренней иерархией. Отдельная таблица `club_ratings`.
- **Тренер-вью**: тренер клуба видит своих учеников, их результаты, рекомендует турниры. Роль `coach` в `club_members`.
- **Страница «Наши чемпионы»**: галерея победителей клубных турниров за историю.

### 5.2. Структура участников и доступ к турниру

- **Категории**: возраст (U10/U12/U14/U16/U18/seniors 35+/45+/55+), пол (муж/жен/микст), уровень (NTRP 2.5/3.0/3.5/4.0/4.5, UTR). В турнире может быть несколько категорий одновременно — нужны «группы регистрации».
- **Посев по уровню**, а не только рандом/ручной. Новый `draw_mode = 'skill_seeded'`: топ-4 по уровню разносятся в разные четверти.
- **Waiting list + auto-promote**: когда лимит мест исчерпан, новые заявки попадают в очередь. При отказе одобренного — первый в очереди автоматически получает место.
- **Частичная регистрация**: игрок бронирует место на 24 часа до оплаты. Если не оплатил — слот возвращается.
- **Приватные турниры по invite-коду** или **email-allowlist**: для закрытых клубных событий и корпоративов.
- **Минимальный/максимальный возраст** в форме регистрации (server-side валидация по дате рождения).

### 5.3. Расписание и корты (детализация сверх FEATURE_RESEARCH)

- **Несколько кортов с покрытием** (hard/clay/grass/indoor) и ценой аренды. Таблица `courts (club_id, name, surface, hourly_rate)`.
- **Блокировка слотов**: корт занят (аренда), погода, ремонт. В UI расписания — затемнённые ячейки.
- **Автораспределение матчей по кортам** с ограничениями: отдых игрока ≥30 мин между матчами, равномерная нагрузка на корты. SQL-функция `auto_schedule_matches(tournament_id, constraints)`.
- **ETA следующего матча** для игрока: при завершении предыдущего матча пушит «на корт через 15 минут». Требует scheduled_at + текущее состояние предыдущих матчей.
- **Перенос матча (reschedule)** с уведомлением обеим сторонам. Аудит истории переносов.
- **Календарь клуба на месяц**: все турниры клуба в ics-формате с экспортом в Google/Apple Calendar.

### 5.4. Финансы и операции

- **Возврат средств** при отмене турнира или за X часов до старта (есть в FEATURE_RESEARCH #14 в общем виде, но без логики возвратов).
- **Split-оплата для пар**: оба игрока оплачивают по 50%, автоматическое подтверждение пары только после обоих платежей.
- **Промокоды** (`EARLY_BIRD_20`) и **ранние регистрации** со скидкой до определённой даты.
- **Дашборд выручки клуба**: турниры, участники, средний чек, повторные плательщики.
- **Invoice / чек** участнику на email (PDF с реквизитами клуба).
- **Множественные методы оплаты**: Stripe (международные), + локальные (Revolut, Paysera для LT, СБП для RU).

### 5.5. Зрители, маркетинг, вовлечение

- **OG-теги для соцсетей**: при шаринге ссылки `/tournaments/:slug` превью показывает название, сетку, даты. Сейчас — голый заголовок.
- **Highlights-лента** на странице клуба: «вчера Иванов обыграл Петрова 6:4 6:3» автоматически из завершённых матчей.
- **Фото-галерея турнира**: upload фотографий организатором или игроками, GDPR-согласие участников при регистрации.
- **Стриминг-оверлей**: URL с чистым HTML-счётом для OBS (OBS browser source). Комментаторы и клубные стримы на YouTube.
- **Sponsors block**: логотипы спонсоров с ссылками на публичной странице турнира. Монетизация для организатора без нагрузки на платформу.
- **Комментарии / чат на странице матча** (опционально) — для вовлечения зрителей в live-трансляциях.

### 5.6. Операционка организатора

- **Template турнира**: «сохранить как шаблон» → «duplicate». Для клубов, проводящих регулярные турниры (каждая суббота).
- **Массовые действия**: одобрить N заявок, отклонить всех из чёрного списка, rebroadcast в Telegram, экспорт только выбранных матчей.
- **Audit log**: кто и когда поменял счёт/сетку/статус. Критично для разрешения споров между игроками.
- **Draw ceremony mode**: анимация жеребьёвки на большой экран (полноэкранный режим в браузере). Для клубных вечеров.
- **Player-reported scores**: игроки подтверждают счёт через ссылку из уведомления. Снимает нагрузку со счётчика. Нужна верификация от обеих сторон матча.
- **Автоматические «ban list»**: игрок, который дважды не явился на турниры клуба, попадает в список без автосогласования.

### 5.7. Надёжность, безопасность, ops

- **2FA для super-admin** (`platform_admins` сейчас только за OAuth — один взлом Google и вся платформа под угрозой).
- **Audit log super-admin**: одобрение/отклонение клубов, кто нажал, когда, причина.
- **Rate limiting публичной регистрации** — см. секцию 3.5.
- **Soft-delete турниров** с восстановлением в течение 30 дней. Сейчас удаление — hard. Случайный клик владельца — и турнир потерян.
- **Backup / export данных клуба**: кнопка «скачать всё» (JSON + CSV заявок + PDF сеток) для клубов, уходящих с платформы. GDPR-дружественно, снимает психологический барьер регистрации.
- **Sentry или аналог**: централизованный сбор frontend-ошибок. Сейчас их не видно.

## 6. Пересмотр приоритетов на 3–6 месяцев (гибридная модель)

FEATURE_RESEARCH.md предлагает 4-фазный план. Ниже — его пересмотр с учётом:
- Расхождений, обнаруженных в секции 2 (P0-долг нужно закрыть раньше).
- Выбора гибридной модели «клубы + индивидуалы».
- Реальной B2B-ценности (клубы платят за то, что удерживает игроков).

### Месяц 1 — Фундамент (P0 + базовые ожидания)
- **P0-долг** из секции 3: live-scoring, `tournaments.club_id`, `players`-сущность, синхронизация CLAUDE.md, rate limiting.
- **Walkover / неявка** (кнопка в ScoreEditor).
- **Дата / время / место** в форме турнира.
- **Light theme** (CSS-переменные уже есть).
- **CSV-экспорт** заявок и расписания.
- **Публичная страница клуба** `/clubs/:slug`.

### Месяц 2–3 — Форматы и клубная ценность
- **Round-robin** (Фаза 2 из FEATURE_RESEARCH).
- **Group stage + playoff** (нет в FEATURE_RESEARCH, но самый востребованный в клубах).
- **Категории участников**: возраст, пол, уровень.
- **Waiting list + auto-promote**.
- **Расписание с кортами**, ETA и email за 15 мин до матча.
- **Telegram-бот** (K7 из FEATURE_RESEARCH — самая дешёвая killer-фича для RU/LT/EE).
- **Email-уведомления** базовые (регистрация, одобрение, начало матча).

### Месяц 4–6 — Удержание и монетизация
- **Профили игроков** + **ELO** (глобальный и клубный).
- **Серии турниров** (K4).
- **Court Mode** для ввода счёта на корте (K5) — большие кнопки, вибрация, минимум шагов.
- **Entry fee** через Stripe + возврат + промокоды.
- **Embed-виджет** сетки + **OG-теги**.
- **Ladder / Box league** (отдельный эпик для клубов, удерживающая фича).
- **Template турнира** и **draw ceremony mode**.

### Отложено за горизонт (осознанно)
- **Double elimination, Swiss, compass draw** — нишевые форматы, спрос низкий.
- **Offline PWA (K2)** — дорого (Service Worker + IndexedDB + очередь Supabase), окупаемость неочевидна.
- **Геймификация / бейджи (K6)** — без критической массы игроков бессмысленна.
- **Bulk CSV-импорт** — нужен 1–2 клубам из 100, делается в обход (руками через форму).
- **Умный сидинг по истории (K3)** — требует >1000 матчей в базе для статистической значимости.

## 7. Риски и допущения

- **Гибридная модель удвоит сложность UI**. Две навигации (клубная vs индивидуальная), две витрины, разные permission-модели. Нужно решение: видит ли non-club организатор ту же страницу турнира, что клубный, или разные. Рекомендую — одна страница, с опциональным блоком «Этот турнир проводит клуб [X]» сверху.
- **Миграция на `club_id` обязательным**. Существующие турниры без клуба — как их трактовать? Варианты: (а) оставить `club_id` nullable для legacy; (б) создать «virtual club» для каждого индивидуального организатора.
- **Stripe = compliance**. KYC клубов, налоговая отчётность по юрисдикциям (LT, RU, EE), возвраты по GDPR/PCI. Это полноценный квартал работы, не неделя.
- **ELO без истории = мусор**. Первые 5–10 матчей игрока рейтинг неинформативен. Нужен период «prov.» (provisional) и флаг в UI.
- **Telegram-бот требует модерации**. Спам от ботов к организаторам — реальный риск. Нужен rate limiting и блок-лист.
- **Live-scoring и offline несовместимы** в MVP. Сначала стабильный online, потом Service Worker.
